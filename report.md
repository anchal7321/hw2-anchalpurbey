# Report: Meeting Transcript Summarizer

## Business use case
For this assignment, I built a simple GenAI workflow that takes AI-transcribed meeting transcripts and turns them into two outputs: a brief summary of important points to consider and a list of actionable next steps. The intended user is a project manager who needs to quickly translate meeting discussion into usable follow-up notes that can help guide project direction. I chose this workflow because it is a realistic business writing task that happens often in project-based work, and even a first draft can save time, improve consistency, and reduce the effort required after meetings.

The input to the system is a meeting transcript, and the output is a short business-style summary plus action items. That makes the task narrow enough for a simple prototype but still useful enough to automate partially. The assignment also asked for a small stable evaluation set, so I created five representative test cases: two normal business cases, one edge case with uncertainty, one sensitive case that clearly requires human review, and one hallucination-risk case involving uncertain numbers and dates. This helped me test the system in a fair and repeatable way rather than relying on a single successful example.

## Model choice
I used Google AI Studio with the 'google-genai' Python SDK, which matched the recommended default in the assignment. My Python app was built to run from the command line, take a selected test case, apply a prompt version, make a real Gemini API call, print the result in structured sections, and save the output to a file. This aligns with the assignment requirement to build a small reproducible Python prototype that uses a real LLM API call and supports prompt variation.

During testing, I tried more than one Gemini model and ran into practical issues. Some earlier keys had quota problems, and 'gemini-2.5-flash' was sometimes unavailable because of temporary overload. After several attempts, 'gemini-2.5-flash-lite' gave the most reliable path for my later test runs, so I used that model for the final testing that informed my report.

I also liked Gemini for this assignment because the setup was straightforward once dependencies were installed: the script used 'google-genai', loaded the API key from the environment, and could be run through 'uv' or Python directly.

## Baseline vs. final design
My baseline prompt was simple: it asked the model to read the meeting transcript and produce a brief summary of important points plus actionable next steps. This baseline often worked reasonably well on straightforward cases. For example, in the supplier and warehouse case, the baseline correctly identified the reduced delivery delay, the finance approval, and the need for warehouse confirmation. However, the baseline also showed why prompt iteration mattered. In some cases, it produced action items that went beyond the transcript and sounded more confident than the original notes supported.

Revision 1 added clearer rules: do not invent facts, do not invent names, dates, owners, or numbers, and explicitly state when information is uncertain. This improved the model's faithfulness. In the supplier case, Revision 1 made the unresolved warehouse confirmation more explicit by labelling it as uncertain rather than simply treating it as a normal next step. In the confidential leadership case, Revision 1 was a much stronger improvement over the baseline because it stopped inventing unsupported actions like preparing future data or monitoring internal leaks and instead stayed much closer to what was said in the transcript.

Revision 2 added an additional safeguard: if the content was sensitive, confidential, or based on unfinished decisions, the model should indicate that human review was needed. This was the most important improvement for the highest-risk case. In the leadership transcript, the final version labelled the layoffs and acquisition target as under consideration and unconfirmed, noted that no decisions were finalized, and explicitly flagged the output for human review. Compared with the baseline, this was a major shift in behaviour: the model moved from generating speculative operational next steps to recognizing that the right response in a sensitive case is caution and escalation, not confident action.

Overall, the final design was better than the baseline in three ways. First, it was more faithful to the transcript and less likely to invent unsupported actions. Second, it handled uncertainty more explicitly by labelling unconfirmed issues rather than blending them into normal summary language. Third, it treated sensitive content more appropriately by adding a human-review boundary.

## Where the prototype still fails or requires human review
Even after prompt revision, the prototype still has clear limitations. The first limitation is overreach. In more ambiguous cases, the model can still suggest action items that sound reasonable but are not strictly stated in the transcript. I saw this in earlier outputs where the system wanted to add ownership, deadlines, or strategic next steps even when the input did not clearly support them. That is a real risk, because in business settings a polished but unsupported recommendation can be more dangerous than an obviously poor output.

The second limitation is sensitivity and organizational judgment. In a confidential leadership case, the system should not behave like a normal summarizer. The final prompt improved this by requiring human review, but the fact that I had to explicitly add that rule shows that the model does not naturally understand organizational risk boundaries on its own. Without that extra instruction, it was willing to generate overly confident or speculative next steps. That means human review is still necessary whenever a transcript contains unfinished decisions, sensitive topics, unclear timelines, or potential personnel consequences.

## Deployment recommendation
I would recommend this workflow only as a "drafting assistant with human review", not as a fully automated business process. It is useful for speeding up the conversion of transcripts into a readable first draft, and it performs reasonably well on routine operational updates where the facts are straightforward and the stakes are low. In those cases, a project manager could save time by using the output as a starting point and then editing it before sharing.

I would not recommend deploying this system in a fully automated way for sensitive or ambiguous cases. If the transcript includes confidential leadership discussion, personnel implications, unclear numbers, or unconfirmed decisions, the output should always be reviewed by a human before it is used or distributed. In my view, the best deployment condition would be an internal support tool where every generated summary is reviewed, edited, and approved by the project manager. Under those conditions, the workflow has value. Without those controls, the risk of overreach, invented detail, or inappropriate framing is too high.
